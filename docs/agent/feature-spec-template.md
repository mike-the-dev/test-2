# Feature Spec: <Feature Name>

## 1) Goal / Outcome
- One sentence that describes the user value.
- Example: "Allow merchants to upload product images and manage them in the admin UI."

## 2) Scope (In / Out)
### In Scope
- Bullet list of what is included.

### Out of Scope
- Bullet list of what is explicitly not included.

## 3) User Stories
- As a <role>, I want <capability> so that <benefit>.
- Keep each story one sentence.

## 4) Acceptance Criteria (Must Pass)
- Bullet list of pass/fail criteria.
- Avoid vague language; measurable only.

## 5) UX / UI Notes
- If UI exists: describe layout, states, and interactions.
- If no UI: state "N/A".

## 6) Data / API Contracts
### Frontend Inputs
- Data required by UI (fields + types).

### Backend Inputs
- API endpoints to add/modify (method + route).
- Request body / query parameters.
- Response shape.

### Validation Rules
- Exact validations (e.g., file size limit, formats).

## 7) Permissions / Auth
- Who can access?
- Any role restrictions?

## 8) Error Handling
- Expected error cases and user-facing messages.
- Backend error codes if applicable.

## 9) Analytics / Logging (if required)
- Event names + properties.
- Logging expectations.

## 10) Tests Required
- Unit tests needed (what behaviors).
- Integration/e2e tests needed.

## 11) Migration / Backfill (if needed)
- Any data migrations or defaults?

## 12) Dependencies / External Services
- Third-party services or internal services.
- Env vars required.

## 13) Edge Cases / Constraints
- Known tricky scenarios.

## 14) Delivery Notes
- Rollout or feature flag?
- Any coordination needed (frontend/backend order).
